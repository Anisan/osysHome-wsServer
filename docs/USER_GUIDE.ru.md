# wsServer - Руководство пользователя

## 1. Назначение модуля

`wsServer` это системный модуль, который поднимает WebSocket-соединение на базе `Flask-SocketIO` и даёт приложению обмен данными с браузером в реальном времени.

Модуль нужен для задач, где обычного обновления страницы недостаточно:

- мгновенно показывать изменения свойств объектов;
- перерисовывать HTML объектов без перезагрузки страницы;
- отправлять уведомления и сообщения пользователю;
- запускать звуки в браузере;
- вызывать методы объектов из клиента;
- передавать произвольные данные из серверных модулей в веб-интерфейс;
- отслеживать активные подключения и их подписки.

## 2. Что получает пользователь

После загрузки страницы фронтенд подключается к Socket.IO-серверу и автоматически:

- подписывается на служебные события `say`, `playsound`, `notify`;
- ищет на странице элементы с привязками свойств и объектов;
- оформляет подписки на нужные свойства и объекты;
- передаёт серверу текущий URL страницы.

За счёт этого пользователь видит обновления сразу, без ручного обновления браузера.

## 3. Требования и условия работы

- пользователь должен быть аутентифицирован;
- соединение создаётся через общий клиентский скрипт `app/templates/includes/websockets.html`;
- модуль работает через `Socket.IO`, поэтому транспорт может быть как `websocket`, так и fallback на long polling;
- для корректного отображения времени используется часовой пояс пользователя, если он задан в свойстве `<username>.timezone`.

Если пользователь не авторизован, подключение отклоняется.

## 4. Административный интерфейс

Страница администратора доступна по адресу:

```text
/admin/wsServer
```

В модуле есть три вкладки.

### 4.1. Clients

Показывает все текущие подключения:

- имя пользователя;
- IP-адрес;
- `SID` сессии;
- время подключения;
- активный транспорт;
- текущую страницу;
- трафик `sent/recv`;
- типы подписок:
  - свойства;
  - объекты;
  - действия;
  - данные.

### 4.2. Monitoring

Позволяет наблюдать WebSocket-события в реальном времени:

- пауза и возобновление лога;
- автоматическое появление фильтров по типу события;
- просмотр последних сообщений в памяти страницы;
- визуальное различение событий по цветам.

### 4.3. Settings / Настройки

Во вкладке `Settings` доступны параметры оптимизации рендера:

- `Enable object render debounce` - включает/выключает серверный debounce для `changeObject`;
- `Object render debounce (ms)` - окно debounce в миллисекундах;
- `Enable property change debounce` - включает/выключает серверный debounce для `changeProperty`;
- `Property change debounce (ms)` - окно debounce в миллисекундах.

Значения по умолчанию:

- включено: `true`;
- задержка: `120 мс`.

Для debounce свойств:

- включено: `false`;
- задержка: `80 мс`.

Рекомендуемый диапазон: `80-300 мс`.

Рекомендуемый диапазон для debounce свойств: `40-150 мс`.

Как это влияет на поведение:

- `changeProperty` может коалесcиться по ключу `Object.property`, если debounce свойств включен;
- `changeObject` коалесcится по объекту и отправляется как последнее состояние после debounce;
- первичное значение свойства при `subscribeProperties` по-прежнему отправляется сразу (`sendProperty(...)`).

## 5. Виджет модуля

Виджет `wsServer` показывает краткую сводку по подключённым клиентам и обновляется через событие `clients`.

Он полезен для:

- дашбордов;
- системных страниц;
- быстрого контроля, подключён ли пользователь;
- проверки, какие подписки использует клиент.

## 6. Автоматические привязки на странице

Модуль умеет обновлять DOM по соглашениям в шаблонах.

### 6.1. Привязка свойства по `id`

Если на странице есть элемент:

```html
<span id="prop:Light1.state"></span>
```

то после подписки сервер будет присылать `changeProperty`, а значение свойства будет подставляться в этот элемент автоматически.

Дополнительно поддерживаются связанные поля:

```html
<span id="prop_changed:Light1.state"></span>
<span id="prop_source:Light1.state"></span>
```

### 6.2. Привязка объекта по `id`

Если на странице есть контейнер:

```html
<div id="obj:Light1"></div>
```

то при событии `changeObject` его HTML будет перерисован через `morphdom` без полной перезагрузки страницы.

### 6.3. Атрибутные привязки

Поддерживаются декларативные привязки:

```html
<div data-prop-display="Light1.online"></div>
<span data-prop-text="Weather.outdoorTemp"></span>
<input data-prop-value="Thermostat.targetTemp">
<input type="checkbox" data-prop-checked="Light1.enabled">
<a data-prop-attr-title="Light1.status"></a>
<img data-prop-attr-src="Camera1.preview">
```

Поддержка работает автоматически после вызова `subscribe()`.

## 7. Пользовательские сценарии

### 7.1. Отображение живого значения свойства

Используйте:

```html
<span id="prop:Sensor1.temperature"></span>
```

или:

```html
<span data-prop-text="Sensor1.temperature"></span>
```

### 7.2. Динамическое скрытие и показ блока

```html
<div
  data-prop-display="Pump1.enabled"
  data-prop-display-true="block"
  data-prop-display-false="none">
  Насос включен
</div>
```

### 7.3. Живое обновление HTML объекта

```html
<div id="obj:Heater1"></div>
```

Когда объект будет перерендерен на сервере, клиент получит новое содержимое автоматически.

### 7.4. Получение системных уведомлений

Клиент по умолчанию подписывается на:

- `say`;
- `playsound`;
- `notify`.

Поэтому сообщения, системные уведомления и звуки приходят без отдельной ручной настройки на большинстве страниц.

## 8. Пример клиентского подключения

```javascript
const socket = io();

socket.on('connect', () => {
  socket.emit('subscribeProperties', ['Light1.state']);
  socket.emit('subscribeObjects', ['Light1']);
  socket.emit('subscribeActions', ['say', 'notify', 'playsound', 'executedMethod']);
  socket.emit('subscribeData', ['telemetry']);
});

socket.on('changeProperty', (data) => {
  console.log('Property changed', data);
});

socket.on('changeObject', (data) => {
  console.log('Object HTML updated', data.object);
});

socket.on('telemetry', (data) => {
  console.log('Telemetry', data);
});
```

## 9. Рекомендации по использованию

- используйте `subscribeProperties` для точечных обновлений значений;
- используйте `subscribeObjects`, когда нужен пересчёт и перерисовка целого HTML-блока;
- не подписывайте клиентов на `*`, если это не экран мониторинга;
- для серверных уведомлений предпочитайте `say` и `notify`, а не произвольные события;
- для модульных потоков данных используйте отдельные имена каналов в `sendData(...)`.
